# 第3章: はじめてのコミットとプッシュ

> 練習ブランチ：`feature/02-first-commit`

> **補足**：この章では、本文中のブランチ名とファイル名をそのまま練習に使います。まだ提出・マージは行いません。

**前提**：Unit 1「環境構築」が完了していること（Sourcetree でリポジトリをクローン済み）。

## 1. この章のゴール

- 練習用ファイル `first-commit-practice.md` をリポジトリに追加できる
- 編集 → ステージング → コミット → プッシュ を一通りできる
- 履歴（History）を見て「いつ・何をしたか」を説明できる
- こまめにコミットする理由を自分の言葉で言える
- AIで考えた改善を、1つのコミットとして残せる

## 2. 用語

| 用語 | 意味 | たとえ |
|------|------|--------|
| **ステージング** | コミットに入れる変更を選ぶ | 提出トレーに乗せる |
| **コミット** | 保存ポイントを作る | 封筒に入れて封をする |
| **プッシュ** | コミットを GitHub に送る | 封筒を郵送する |
| **プル** | GitHub の変更を PC に取り込む | 届いた郵便を受け取る |

## 3. 今回の流れを先に見る

この章でやることは、次の5ステップです。

1. `develop` から `feature/02-first-commit` を作る
2. 練習用ファイル `first-commit-practice.md` を追加する
3. さらに1か所だけ具体化して、2回目のコミットを作る
4. `History` で履歴を見る
5. GitHub にプッシュする

前章で `develop` を用意したので、今回はその上に **安全な作業台として feature ブランチを1本作る** ところから始めます。  
ブランチの意味そのものは次章で詳しく整理します。

## 4. 事前確認：今いるブランチ

Sourcetree 左のブランチ一覧で **太字/チェックマーク** になっているものが「今いるブランチ」です。

この章では、`main` を直接触らない練習として **`feature/02-first-commit` 上で作業** します。

### 4-1. 最初に `feature/02-first-commit` を作る

1. `develop` が見えるか確認する
2. まだ無ければ、前章の「`develop` を作っておく」を見て先に作成する
3. `develop` をチェックアウトして `Pull`
4. 上部の `Branch` をクリック
5. ブランチ名に `feature/02-first-commit` を入力
6. `Checkout new branch` にチェックを入れて作成

ブランチの役割は次章で詳しく学びます。  
この章では「`main` を直接触らず、練習用の作業台で試す」と理解できれば十分です。

## 5. 操作手順

### 5-1. `first-commit-practice.md` を追加する

お使いのエディタ（例：VS Code）で、リポジトリ直下に `first-commit-practice.md` を作成します。
内容は短いメモで十分です。たとえば次のように書いて保存してください。

```md
# Git練習メモ

- 今日やること: コミットとプッシュを一通り試す
- 気づいたこと: ステージングはコミット前の確認場所
- 次に試したいこと: ブランチを作って安全に作業する
```

この1回目のコミットでは、**まず練習用のファイルをリポジトリに入れる** ことが目的です。
ここではまだ内容を大きく磨き込まなくて大丈夫です。

### 5-2. 追加したファイルをコミットする

- Sourcetree の `File Status` タブを開く
- `first-commit-practice.md` が `Unstaged files` に出ていることを確認
- 差分を確認して `Stage`
- コミットメッセージを書く（例：`first-commit-practice.md を追加`）
- `Commit` をクリック

ここで作る1回目のコミットは、「練習用の内容を Git 管理に乗せた」という意味があります。

### 5-3. 1か所だけ改善する

続けて、`first-commit-practice.md` の内容を **1か所だけ具体化** して保存します。

おすすめの修正例：

- 「今日やること」を、もう少し具体的な手順に書き換える
- 「気づいたこと」を、自分の言葉で1文追加する
- 「次に試したいこと」を、次章でやるブランチ操作に寄せて書き換える

迷ったら AI に次のように聞いてください。

```text
「first-commit-practice.md のこの1行を、初心者でも行動に移しやすい文章に1段具体化して」
```

### 5-4. 2回目の変更を確認する

- Sourcetree の `File Status` タブを開く
- `Unstaged files` に変更ファイルが出ていることを確認
- ファイルをクリックすると差分（diff）が見られる

`Unstaged files` に何も出ないとき：エディタで保存できているか確認 / Sourcetree で正しいリポジトリを開いているか確認。

ここで見ているのは、「これからコミット候補になる変更」です。  
まだ履歴には入っていないので、内容を見直してから次に進めます。

### 5-5. 2回目の変更をステージングする

- `Unstaged files` のファイルをクリックして差分を確認
- `Stage`（＋/チェック）をクリックして `Staged files` に移す
- 「全部まとめてOK」なら "全部ステージ" でも大丈夫

**注意**：パスワード・トークン・個人情報が入ったファイルはステージしない。間違えて入れたら `Unstage` で外す。

### 5-6. 2回目のコミットを作る

- コミットメッセージを書く（例：`練習メモを具体化`）
- `Commit` をクリック

メッセージのコツ：後で見返したとき「何をしたか」が分かる日本語でOK。`update` だけは避ける。

コミットは「作業の区切り」で残すと分かりやすいです。  
たとえば「今日やることを具体化した」と「次に試したいことを整理した」は、できれば分けてコミットします。

コミットできないとき：
- `Staged files` が空 → 先にステージングする
- 「名前/メールが必要」→ 前章の「3-4」で設定する

### 5-7. 履歴（History）を読む

- `History` タブを開く
- `first-commit-practice.md を追加` と、2回目の改善コミットが並んでいることを確認
- コミットをクリックすると変更内容が見られる

この章では、「追加」と「改善」を分けて履歴に残せていれば十分です。

### 5-8. プッシュする

- 上部の `Push` をクリック
- 送るブランチにチェックが入っていることを確認
- `Push` をクリック

この章では **`feature/02-first-commit` にチェックが入っていること** を確認してください。

プッシュ後、GitHub のリポジトリページを開いて、`first-commit-practice.md` の変更が反映されていることを確認しましょう。

## 6. 余裕があれば：`.gitignore` で余計なファイルを防ぐ

`.gitignore` は「コミットしないファイル一覧」です。リポジトリ直下に作成します。

```gitignore
.DS_Store
node_modules/
.env
```

作成したらステージング → コミット → プッシュしてOKです。

最初は必須ではありません。  
意味が分かるようになってから追加しても大丈夫です。

## 7. やり直しの基本

| やりたいこと | 操作 |
|-------------|------|
| ステージングを戻す | `Staged files` のファイルを右クリック → `Unstage` |
| 変更を捨てる（※戻せない） | `Unstaged files` のファイルを右クリック → `Discard` |
| 直前のコミットメッセージを直す（未プッシュ） | コミット画面で `Amend last commit` にチェック |
| プッシュ済みのコミットを取り消す | `History` → 対象を右クリック → `Reverse commit` / `Revert` |

`.env` をプッシュしてしまった場合は、メンターに相談してください（秘密情報の無効化が必要です）。

## 8. AIに聞いてみよう

```text
「Staged files に入っているのは（ファイル名）。このままコミットしてOK？」
「Commit が押せない。確認する順番を教えて」
「History のこのコミットを取り消したい。安全な方法は？」
「first-commit-practice.md のこの1行を1文だけ良くしたい。差分が小さくなる改善案を1つだけ出して」
```

## 9. ハンズオン課題

1. `first-commit-practice.md` を追加してコミットする
2. 1つ目の改善を入れて、2回目のコミットを作る
3. 別の観点をもう1か所改善して、3回目のコミットを作る
4. GitHub のリポジトリページで、履歴が3つに分かれて見えることを確認する
5. 余裕があれば GitHub で `feature/02-first-commit` → `develop` の PR 作成画面だけを開き、次章以降の流れを予習する（まだ提出・マージはしない）

**合格条件**：GitHub 上でコミット履歴が確認できる。
