# なぜGitを学ぶのか

> **この章は「読むだけ」でOKです。**
> 前章で確認した「自分のエンジニア設計書」を、Git学習に接続します。

## 1. この章のゴール

- Git/GitHubを学ぶ目的（提出管理・変更履歴・バックアップ）を理解する
- Git学習の全体フローを把握する
- 最小限の用語（ローカル/リモート/コミット/プッシュ/ブランチ）を押さえる
- 「AIで改善した内容を安全に反映するためにGitが必要」と理解する

## 2. なぜプログラミングの前にGitをやるのか

プログラミング学習では「壊れた」「戻せない」「提出ミス」が頻発します。
Gitを先に学ぶと、次の2点で学習コストを下げられます。

- **提出物の管理**：途中経過と完成形を安全に提出できる
- **バックアップ**：PCトラブル時もGitHubに履歴が残る

このカリキュラムでは CLI ではなく、**Sourcetree（GUI）** を中心に進めます。

さらに今回は、前のAIカリキュラムで作った **自分のエンジニア設計書** を題材に進めます。  
つまり、このカリキュラムでやる Git 操作は「適当な練習」ではなく、**自分の考えを更新した履歴を残す実践**です。

| 道具 | たとえ | 役割 |
|------|--------|------|
| **GitHub** | ネット上の金庫 | 提出先＋バックアップ |
| **Sourcetree** | 金庫を操作するリモコン | クリックでGit操作 |

## 3. 最初に覚える用語（5つだけ）

| 用語 | 意味 |
|------|------|
| **ローカル** | 自分のPC側 |
| **リモート** | GitHub側 |
| **コミット** | 保存ポイント（後で戻れる地点） |
| **プッシュ** | コミットをGitHubへ送ること |
| **ブランチ** | 安全に作業するための分岐コピー |

## 4. 今回の題材：`engineer-design.md`

このGitカリキュラムでは、前のAIカリキュラムで作った最終成果物を、クローンした練習用リポジトリに `engineer-design.md` として配置して使います。

各章でやることは、例えば次のような改善です。

- AI に「この設計書の曖昧な点」を聞く
- 「目指すエンジニア像」を1段具体化する
- 「最初の3か月の行動計画」を整理する
- 変更理由を PR に書いて提出する

これによって、Git を学ぶ理由が次のように具体化します。

- AI に相談して内容を直したら、その差分を履歴として残したい
- 良い修正と失敗した修正を分けて管理したい
- 他人に「何をどう変えたか」を説明したい

## 5. このカリキュラムの進め方

| 順番 | 章 | 内容 | Git操作 |
|------|-----|------|---------|
| 1 | 前章（Unit 0） | キャリア設計・学習設計 | なし |
| 2 | **この章** | Gitを先に学ぶ理由と全体像 | なし（読むだけ） |
| 3 | Unit 1 | 環境構築（GitHub/Sourcetree） | 設定のみ |
| 4 | Unit 2 | コミット・ブランチ・PRの基本 | 操作開始 |
| 5 | Unit 3 | 課題提出ワークフローを実践 | 実践 |
| 6 | Unit 4 | トラブル時の参照 | 参照用 |

## 6. 提出フロー（プレビュー）

1. `develop` から `feature/課題名` を作る
2. `engineer-design.md` をAIと一緒に改善する
3. 変更をコミットしてプッシュする
4. GitHubでPR（Pull Request）を作る
5. マージ後にローカルを最新化する

詳細は Unit 2〜3 で実践します。

## 7. 運用ルール（Gitflow 簡易版）

| ブランチ | 役割 |
|----------|------|
| `main` | 最終成果物（通常は直接触らない） |
| `develop` | 開発の基準線 |
| `feature/課題名` | 個別課題の作業ブランチ |
