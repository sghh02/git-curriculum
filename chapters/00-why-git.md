> **この章は「読むだけ」でOKです。** 環境構築はまだなので、Git操作は次章から始めます。

## 1. この章のゴール

## 2. なぜプログラミングの前にGitをやるのか

プログラミング学習では「動かない」「壊れた」「どこを直せばいいか分からない」が日常的に起きます。Git を先に学ぶと、次の2つが一気にラクになります。

- **提出物の管理** — 課題の途中経過と完成形を、間違えずに提出できる
- **バックアップ** — PC が壊れても GitHub（ネット上）に課題が残る

難しい黒い画面（CLI）は使いません。このカリキュラムでは **Sourcetree（GUI）** をメインに使います。

### たとえ話

| 道具 | たとえ | 役割 |
|------|--------|------|
| **GitHub** | ネット上の金庫 | 提出先＋バックアップ |
| **Sourcetree** | 金庫を操作するリモコン | クリックで操作できる |

## 3. 最初に覚える用語（5つだけ）

| 用語 | 意味 |
|------|------|
| **ローカル** | 自分の PC 側 |
| **リモート** | GitHub 側 |
| **コミット** | 保存ポイント（あとで戻れる地点） |
| **プッシュ** | コミットを GitHub に送ること |
| **ブランチ** | 作業用の別コピー（安全に試すための作業台） |

## 5. 課題提出の全体像（プレビュー）

この先ずっと使う「型」です。**今は「こういう流れなんだな」と眺めるだけでOKです。**

1. `develop` から `feature/課題名` を作る
2. エディタで編集して保存する
3. Sourcetree でコミット → プッシュ
4. GitHub で PR（Pull Request）を作る
5. マージ後、ローカルを最新化

各ステップは Unit 2〜3 で丁寧に学びます。

## 6. 運用ルール（Gitflow 簡易版）

すべてのカリキュラムで次の3本のブランチを使います。

| ブランチ | 役割 |
|----------|------|
| `main` | 最終成果物（基本は触らない） |
| `develop` | 開発のベース（課題を統合する枝） |
| `feature/課題名` | 自分の作業用（ここからPRを出す） |
