# なぜGitを学ぶのか

> **この章は「読むだけ」でOKです。**
> 前章で確認した「自分のエンジニア設計書」を、Git学習に接続します。

## 1. この章のゴール

- Git とは何かを、初心者向けに説明できる
- Git / GitHub / Sourcetree の役割の違いを理解する
- Git を先に学ぶ理由（提出管理・変更履歴・バックアップ）を理解する
- 「AIで改善した内容を安全に反映するためにGitが必要」と理解する

## 2. まず、Gitとは何か

Git は、**ファイルの変更履歴を記録し、あとで見返したり戻したり比較したりできる仕組み**です。  
専門的には「バージョン管理システム」と呼ばれます。

コード専用の道具ではありません。  
このカリキュラムで扱う `engineer-design.md` のような Markdown ファイルでも使えます。

プログラミング学習では、こんなことがよく起きます。

- どこを直したら動かなくなったか分からない
- 昨日までの状態に戻したい
- 先生やメンターに「何を変えたか」を見せたい
- PC が壊れても提出物を残したい

Git は、こうした問題に備えるための道具です。

特に大事なのは、Git がただの「上書き保存」ではないことです。  
**その時点の状態を履歴として残していける**ので、あとから「どの時点で何を変えたか」を追えます。

イメージとしては、ファイルを毎回「写真」のように残していく感覚です。

1. 1回目のコミットで、最初の状態を残す
2. 2回目のコミットで、改善後の状態を残す
3. 2つを見比べて「どこを直したか」を確認できる

この感覚を持つと、コミットや履歴がかなり理解しやすくなります。

## 3. Git / GitHub / Sourcetree の違い

初心者が最初に混ざりやすいのが、この3つです。

| 名前 | 役割 | たとえ |
|------|------|--------|
| **Git** | 変更履歴を記録する仕組みそのもの | 保存履歴のルール |
| **GitHub** | Git で管理した内容をネット上に置く場所 | ネット上の保管庫 |
| **Sourcetree** | Git を画面操作で使うためのツール | 保管庫を扱う操作パネル |

このカリキュラムでやることを一言でいうと、

- **Git で履歴を作る**
- **GitHub に送って保存・提出する**
- **Sourcetree でそれを分かりやすく操作する**

という流れです。

## 4. Gitでできること

Git を使うと、主に次のことができます。

- **履歴を残す**：いつ何を変えたかを記録できる
- **比較する**：どこが変わったかを差分で見られる
- **戻す**：調子が悪くなったら前の状態を参照できる
- **分けて試す**：ブランチを使って安全に別案を試せる
- **共有する**：GitHub に送って提出・レビューしやすくなる

つまり Git は、「ファイルを保存する道具」というより、**変更を安全に管理する道具**です。

## 5. なぜプログラミングの前にGitをやるのか

プログラミング学習では「壊れた」「戻せない」「提出ミス」が頻発します。  
Git を先に学ぶと、技術学習そのもののコストを下げられます。

- **提出物の管理**：途中経過と完成形を分けて扱える
- **バックアップ**：PC トラブル時も GitHub に履歴が残る
- **やり直しやすさ**：失敗しても前の状態を基準にできる
- **説明しやすさ**：先生やメンターに「何を変えたか」を伝えやすい

このカリキュラムでは CLI ではなく、**Sourcetree（GUI）** を中心に進めます。  
最初からコマンドを暗記するのではなく、まずは「履歴を安全に扱う感覚」をつかむことを優先します。

## 6. 最初に覚える用語（5つだけ）

| 用語 | 意味 |
|------|------|
| **ローカル** | 自分の PC 側 |
| **リモート** | GitHub 側 |
| **コミット** | その時点の状態を履歴として残すこと |
| **プッシュ** | ローカルのコミットを GitHub に送ること |
| **ブランチ** | 安全に別の作業を進めるための作業ルート |

今は厳密に覚え切らなくて大丈夫です。  
このあと実際に操作しながら、意味がつながっていきます。

## 7. 今回の題材：`engineer-design.md`

この Git カリキュラムでは、前の AI カリキュラムで作った最終成果物を、クローンした練習用リポジトリに `engineer-design.md` として配置して使います。

各章でやることは、例えば次のような改善です。

- AI に「この設計書の曖昧な点」を聞く
- 「目指すエンジニア像」を1段具体化する
- 「最初の3か月の行動計画」を整理する
- 変更理由を PR に書いて提出する

この題材を使うと、Git を学ぶ理由がかなり具体的になります。

- AI に相談して内容を直したら、その変化を履歴として残したい
- 良い修正と失敗した修正を分けて管理したい
- 他人に「何をどう変えたか」を説明したい

つまり、このカリキュラムでやる Git 操作は「適当な練習」ではなく、**自分の考えを更新した履歴を残す実践**です。

## 9. 提出フロー（プレビュー）

1. `develop` から `feature/課題名` を作る
2. `engineer-design.md` を AI と一緒に改善する
3. 変更をコミットしてプッシュする
4. GitHub で PR（Pull Request）を作る
5. マージ後にローカルを最新化する

詳細は Unit 2〜3 で実践します。

## 10. 運用ルール（Gitflow 簡易版）

| ブランチ | 役割 |
|----------|------|
| `main` | 最終成果物（通常は直接触らない） |
| `develop` | 開発の基準線 |
| `feature/課題名` | 個別課題の作業ブランチ |
